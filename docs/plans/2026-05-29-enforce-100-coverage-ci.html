Implementation Plan · 2026-05-29

Enforce 100% test coverage in CI with merge protection

Planning 0 / 0 tasks 0 / 0 steps TDD

Architecture

Add pytest-cov and measure line coverage both locally and in the existing CI job. Add `# pragma: no cover` to the only two unreachable entrypoint lines (the blocking app.run_polling() in bot.py and the if __name__ == "__main__" guard), then write tests that close every remaining gap so coverage hits a literal 100%. Finally flip on `--cov-fail-under=100` so CI goes red below 100%, and add a GitHub branch-protection ruleset on main that requires the CI check, hard-blocking the merge button.

  3. Create a branch-protection ruleset on main requiring the CI status check, via gh api --method POST repos/aliir74/toop/rulesets with a required_status_checks rule whose context is lint + typecheck + tests, pull_request rule enabled, target ref ~DEFAULT_BRANCH
  4. Verify the rule is active: gh api repos/aliir74/toop/rulesets lists it, and PR #1 shows the check as Required

Rationale

CI red alone does not block the merge button — a branch-protection ruleset requiring the check is what hard-blocks merge, per the user's decision.

Risks
  • If the repo is private on a free plan, classic branch protection can be limited; rulesets are the modern path and generally available — if the API rejects, report the exact error rather than silently skipping
  • The required-check context string must exactly match the job name `lint + typecheck + tests`, not the workflow name `CI`
